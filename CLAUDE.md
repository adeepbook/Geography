# CLAUDE.md — 地理纵深项目

> 本文件是项目常驻上下文。每次会话开始请先读它,按其中的架构、数据格式、纪律与工作方式行事。

## 1. 项目是什么
一张可点击的世界地图:点开任何一个地方,它从平面符号展开成一个有纵深、有来历的真实对象——既能往下拆(天上/地面/地下),也能往回看(怎么形成、形成之前是什么)。
核心价值两个词:**真实 + 理解**。它不是查询工具,是给真正喜欢地理的人"看穿一个地方"用的。

## 2. 架构(5 模块,只靠固定接口相连)
- **M1 世界地图**:入口 + 作者标注工具。
- **M2 地点库**:每个地点是一份"脚本"(模块组合 + 本地参数 + 薄注解)。
- **M3 过程库**:有限、写定、全球复用的"形成机制"模块。
- **M4 知识卡**:把脚本渲染成卡片。
- **M5 数据层**:气候/地形/土壤等真实数据,贯穿全部。

接口铁律(改任一模块不应牵动其他):
- M2→M3 只能通过「模块ID + 参数」引用,不得在地点里自写机制内容。
- M4 只认 M2 的脚本格式,不关心内容来自哪。
- M1 只存「地点ID + 坐标 + 类型 + 状态」,不存内容。

## 3. 技术栈
- 前端:TypeScript + React + Vite
- 世界地图:MapLibre GL JS(开源免费)
- 剖面图与动画:SVG + D3
- 三档 3D:Three.js(**后期再做,现在不碰**)
- 内容:Git 仓库里的 YAML / Markdown(**不进数据库**)
- 数据加工:Python(GDAL/rasterio)**离线预处理**,不在 app 内
- 部署:纯静态站(GitHub/Cloudflare/Vercel Pages)

## 4. 内容数据格式(代码要照此解析)
内容文件已先于代码存在,放在 `/content`。加载器要能直接吃下它们:

**地点脚本** `/content/locations/*.yaml`(可能是多文档 YAML,用 `---` 分隔,需逐文档解析)
字段:`地点ID, 名称, 坐标[lat,lng], 区位, 类型标签[], 一句话, 形成时间线[{事件,模块,参数}], 垂直拆解{天上[],地面[],地下[]}, 本地注解, 事实来源[], 状态`

**过程模块** `/content/processes/*.md`(目前是"一文件多模块"的 Markdown,用 `##`/`###` 标题分隔每个模块)
需从每个模块小节解析出:`模块ID(形如 外力/流水/下切与峡谷), 一句话, 原理讲解, 剖面画法, 动画脚本, 参数槽[], 易错警示, 状态`;正文保留为 markdown 渲染。
- 地点脚本里「模块」字段的值 = 过程模块的 `模块ID`,据此建立 M2→M3 的链接。
- 类型标签 → 钉子颜色:火山与板块#D85A30 造山与构造#8B6B4E 流水#3E7DBA 冰川#5FB3C9 喀斯特#4E9444 风与干旱#C98A26 气候与生物#1D9E75

## 5. 知识卡结构(M4,所有地点共用一个模板)
1. 头部:名称 / 区位 / 类型标签 / 地图定位
2. 形成时间轴:可拖动滑块 + 播放;配 SVG 剖面随时间线事件变化;往回拨 = "形成之前"
3. 垂直拆解:天上 / 地面 / 地下,逐层可展开,每层挂模块
4. 过程模块芯片:点击进入该模块的通用原理页
5. 来源与精度声明:状态非"已核对"的内容要有小标记

## 6. 视觉三档(逐档升级,不推倒重来)
一档 扁平剖面(快) → **二档 轴测拆解卡(主力,先做扎实)** → 三档 真实3D(Three.js,旗舰地点,后期)。
当前阶段:把二档卡做好;3D 先不做。

## 7. 纪律(最重要)
- **真实第一:绝不在内容里编造地质事实。** 内容只来自人工提供的 YAML/MD;代码里若需占位数据,必须明确标注 sample/placeholder,不得伪装成真实内容。
- 状态机:内容条目 `状态 ∈ {起草, 待核对, 已核对}`;UI 对非"已核对"项显示小标记。
- 不用 AI 生成图当技术内容;高级感来自真实几何(矢量 / DEM)。

## 8. 工作方式(我做你审)
- 重要改动先给计划或 diff,等我确认再落地。
- 分小步、可审阅地推进;每步结束简述"做了什么 + 怎么验证"。
- 代码规范:TypeScript strict;函数式组件 + Hooks;组件小而专;命名清晰;必要处写简短注释。
- 不确定的设计先问我,别擅自扩大范围。
